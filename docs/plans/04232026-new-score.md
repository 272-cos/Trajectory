# Plan: Align per-component minimum with DAFMAN §3.7.4 (chart-floor / `*` row) across all scoring tables

## Context

The repo currently enforces a **60% per-component minimum** for Cardio, Strength, and Core:

```js
// src/utils/scoring/constants.js:70-75
export const COMPONENT_MINIMUMS = {
  [COMPONENTS.CARDIO]: 60.0,
  [COMPONENTS.STRENGTH]: 60.0,
  [COMPONENTS.CORE]: 60.0,
  // BODY_COMP: no minimum per DAFMAN §3.7.1
}
```

No `§` is cited for the 60% figure. Research across the repo's extracted DAFMAN 36-2905 text, AFPC public pages, and secondary summaries (Air & Space Forces Magazine, community PT calculators) could not locate the 60% rule anywhere in DAFMAN 36-2905 (24 March 2026). The authoritative rule is:

> DAFMAN §3.7.4 (`docs/DAFMAN-36-2905.md:1316-1318`): "Completing the minimum exercise repetition/duration in all fitness assessment component modalities does not generate enough points to earn a composite score of 75 or greater. **Repetition/durations below the required minimum receive a component score of zero.**"

> DAFMAN §3.1 (`docs/DAFMAN-36-2905.md:1142-1143`): "Members taking the PFRA must achieve a minimum score in each of the physical components excluding body composition and an overall composite score of ≥ 75."

> DAFMAN §3.7.1 (`docs/DAFMAN-36-2905.md:1268-1271`): "50 points for Cardiorespiratory, 20 points for Body Composition (does not have a minimum requirement), 15 points for Muscular Strength, and 15 points for Core Endurance."

On the AFPC PFRA Scoring Charts (`docs/PFRA-Scoring-Charts.md`), the **asterisk (`*`)** on the lowest printed row of each component table marks "the required minimum repetition/duration" from §3.7.4. Below that row, §3.7.4 awards zero points and the component minimum is unmet. At or above that row, the component minimum IS met; the only remaining way to fail the PFRA is the composite < 75 gate from §3.6.1.

Confidence: **Likely** that no 60% rule exists in DAFMAN 2026. Primary-source PDFs returned 403 during research; the extracted text in `docs/DAFMAN-36-2905.md` and all accessible summaries support the `*`-row interpretation. The 60% constants are almost certainly a legacy assumption from pre-2026 scoring or an initial implementation decision that predates the 2026 regulation. This plan resolves the mismatch, covers all 18 component × bracket tables and the WHtR / walk tables, and touches every engine that reads `COMPONENT_MINIMUMS`.

This is a **scoring-pillar change** per `docs/DECISIONS.md` and `feedback_scoring_pillar_rule.md`. External output (composite, per-component pass/fail, "minimum to pass" UI messaging, PDF scorecard) will change for any user previously caught in the gap between the `*` row and the 60% row. Strength and core users are the most affected (the 60% gate was ~43% stricter than DAFMAN for M/<25 push-ups, for example: 49 reps vs 30 reps). Cardio users are slightly more lenient today (code 60% < chart 70%), so migrating to the chart `*` row makes cardio **stricter** by 5 percentage points.

## The real DAFMAN rule, encoded

| Component | Minimum (DAFMAN §3.7.4) | How to detect programmatically |
|---|---|---|
| Cardiorespiratory (run / HAMR) | `*` row of the chart = 35.0 pts of 50 (verified M/<25: run 19:45, HAMR 42 shuttles) | `externalPoints > 0` for the tested exercise |
| Muscular Strength (push-ups / HRPU) | `*` row = 2.5 pts of 15 (verified M/<25 push-ups: 30 reps) | `externalPoints > 0` |
| Core Endurance (sit-ups / CLRC / plank) | `*` row = 2.5 pts of 15 (verified M/<25 sit-ups: 27, CLRC: 33, plank: 1:35) | `externalPoints > 0` |
| Body Composition (WHtR) | None per §3.7.1 | Always met (when not BFA-gated) |
| 2km Walk | Pass/fail only per §3.7.3 | Existing walk-pass gate |

Observation: because `lookupScore` already returns `external points = 0` for below-chart-floor performances (per kitchen-sink sprint work, see `docs/plans/2026-04-14-kitchen-sink-polish-sprint.md:360-364`), the structural rule collapses to one line:

> **Component minimum is met iff the component is exempt, is BC, is walk, or produced `externalPoints > 0`.**

No percentage arithmetic required. `COMPONENT_MINIMUMS` can be **deleted**.

## Recommended approach

### Step 1 — `constants.js`: retire `COMPONENT_MINIMUMS`, add a semantic helper

`src/utils/scoring/constants.js:70-75`. Delete the `COMPONENT_MINIMUMS` export. Replace with an exported Set of components that have a chart-floor minimum:

```js
// DAFMAN 36-2905 §3.7.4: the minimum repetition/duration is the asterisk (`*`)
// row on the PFRA Scoring Charts. Performances below that row earn zero
// external points and the component minimum is unmet. DAFMAN §3.7.1 explicitly
// exempts Body Composition; §3.7.3 makes 2km walk pass/fail (no point minimum).
export const COMPONENTS_WITH_CHART_FLOOR_MINIMUM = new Set([
  COMPONENTS.CARDIO,
  COMPONENTS.STRENGTH,
  COMPONENTS.CORE,
])
```

No new numbers in the constant. The chart floor IS the table's last row; the tables are the source of truth.

Bump version constants in the same file:
- `CHART_VERSION = '2026-03 PFRA Final'` (replaces `'Sep 2025 Provisional'`, line 192).
- `CHART_RELEASE_DATE = '2026-03-01'` (line 193).
- Add `REGULATION_VERSION = 'DAFMAN 36-2905, 24 March 2026'` (new).

### Step 2 — `scoringEngine.js`: gate on `externalPoints > 0`, not percentage

`src/utils/scoring/scoringEngine.js:217-219`:

```js
// OLD:
const minimum = COMPONENT_MINIMUMS[type] ?? 0
// ... percentage >= minimum
```

becomes:

```js
import { COMPONENTS_WITH_CHART_FLOOR_MINIMUM } from './constants.js'

function componentMinimumMet(componentType, externalPoints) {
  if (!COMPONENTS_WITH_CHART_FLOOR_MINIMUM.has(componentType)) return true
  return externalPoints > 0
}
```

Every callsite that computed `percentage >= minimum` becomes `componentMinimumMet(type, points)`. No more division, no more per-bracket lookup.

### Step 3 — `reverseScoring.js`: two separate functions

`reverseLookup` (line 112-146) stays algorithmic (fix the descending-loop break as in the original hazy-pizza plan). It's still useful for "what performance earns me N external points?"

`getMinimumToPass` (line 282-298) changes semantic: return the `*` row directly for Cardio/Strength/Core; null for BC and walk. Implementation:

```js
export function getMinimumToPass(exercise, ageBracket, gender) {
  const component = EXERCISE_TO_COMPONENT[exercise]
  if (!component) return null
  if (exercise === EXERCISES.WALK_2KM) return null
  if (!COMPONENTS_WITH_CHART_FLOOR_MINIMUM.has(component)) return null

  const table = getScoringTable(gender, ageBracket, exercise)
  if (!table || table.length === 0) return null
  const floor = table[table.length - 1] // the `*` row (lowest threshold, lowest points)
  return {
    threshold: floor.threshold,
    points: floor.points,
    // minimumPct retained for back-compat renderers, computed from the chart floor
    minimumPct: (floor.points / COMPONENT_WEIGHTS[component]) * 100,
    displayValue: formatReverseValue(exercise, floor.threshold),
  }
}
```

For every (exercise, age-bracket, gender) tuple the floor row is `table[table.length - 1]`. Verified for push-ups M/<25 (row 26 = `{30, 2.5}`) and for the cardio/HAMR charts (last row on page 3 = `{19:45, 35.0}` and `{42, 35.0}`).

### Step 4 — Delete the 60% arithmetic from all four engines

| File:Line | Old | New |
|---|---|---|
| `src/utils/scoring/strategyEngine.js:419-420` | `COMPONENT_MINIMUMS[comp] \|\| 60` then `primary.scorePct * 100 < compMinimumPct` | `COMPONENTS_WITH_CHART_FLOOR_MINIMUM.has(comp) && primary.points === 0` |
| `src/utils/recommendations/recommendationEngine.js:549-557` | `COMPONENT_MINIMUMS[comp] ?? 60`, `gapBelowMin`, `isFailing = data.percentage < minPct` | Replace `isFailing` with `data.points === 0 && COMPONENTS_WITH_CHART_FLOOR_MINIMUM.has(comp)`; `gapBelowMin` becomes the gap to the `*`-row point value (not a percent) and only applies to floor components |
| `src/utils/scoring/optimalAllocation.js:234` | `maxPts * (COMPONENT_MINIMUMS[key] / 100)` | `COMPONENTS_WITH_CHART_FLOOR_MINIMUM.has(key) ? getChartFloorPoints(key, exercise, gender, ageBracket) : 0` (helper reads `table[table.length - 1].points`) |
| `src/utils/projection/projectionEngine.js:82, 391` | Same 60% math | Same floor-points replacement |

`belowMinimum` across all engines becomes: "the component is in `COMPONENTS_WITH_CHART_FLOOR_MINIMUM` AND external points = 0." Identical signal; simpler shape.

## Rollout notes

- **S-code compatibility:** existing stored S-codes encode raw performance values, not cached points (per `constants.js:190-192`). All historical S-codes automatically re-score under the new rule the next time a user opens the app. No migration required.
- **Score deltas per user:**
  - Users scoring 2.5-8.5 external points on strength or core **will see their component move from "failing" to "passing"** at the component gate. Their composite score is unchanged; only the pass/fail flag on that component (and possibly the overall) may flip.
  - Users scoring 30-34 external points on cardio **will see their component move from "passing" to "failing"** because the chart `*` row is 35.0 and the old gate was 30.0. This affects an estimated small band of cardio performances just below the chart floor under the prior clamp logic; verify the count by grepping existing tests.
- **PWA cache:** service worker will update on next deploy; no user action required. Static site, no backend state to migrate.
- **Docs:** the DAFMAN-COMPLIANCE-MATRIX.md seed is the right moment to begin the §-to-code index that the pillar rule has been calling for since its inception.

## Out of scope

- Any change to exercise modality, time limits, WHtR thresholds, walk pass/fail rules, BFA gate logic, age-bracket mapping, or S-code / D-code format.
- Creating new UI affordances. This plan updates copy and gate logic; no new screens.
- Changing `internalPoints` math. Internal scoring remains tunable per the pillar rule and is unaffected by this external-side correction.
- Primary-source verification of the DAFMAN PDF requires access the research-agent did not have; if the user can paste §3.7 verbatim from the signed PDF, that promotes this plan from Likely to Confirmed. Not a blocker for implementation, but noted.
